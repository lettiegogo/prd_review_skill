# 产品文档审核工作流 (PRD Review Workflow)

本 Skill 定义了审核产品需求文档 (PRD) 的交互式工作流。核心目标是确保 PRD 符合标准，并通过对话循环完善文档。

## 触发条件
当用户提供飞书文档链接或 PRD 文件，并要求进行“审核”、“评审”或“检查”时触发。

## 工作流步骤

### 阶段 1：读取与预处理
1. **获取文档**: 
   - 如果用户提供飞书链接：使用 `feishu-doc-export` skill 下载文档。
     - 🔴 **关键步骤**：飞书原始导出内容包含大量 XML-like 标签（如 `<lark-table>`, `<image token="...">`），**不可读**。必须使用 `feishu-doc-export` 下的 `scripts/clean_feishu_md.py` 脚本或等效的正则逻辑将其清洗为标准 Markdown，再进行后续处理。
   - 如果用户提供本地文件：直接读取文件内容。
2. **识别需求类型与加载标准**:
   - 分析 PRD 内容，判断需求类型：🔴 全新 (New) / 🟡 迭代 (Iteration) / 🟢 优化 (Optimization)。
   - 读取 `references/ops_web_standard.md` 并加载对应类型的审核章节：
     - **全新需求**: 关注权限、状态机、日志 (高严格度)。
     - **迭代需求**: 关注兼容性、旧数据、在途工单 (中严格度)。
     - **功能优化**: 关注变更范围、性能、回滚 (低严格度)。

### 阶段 2：初次审核与合规检查
1. **执行审核**: 对照标准检查 PRD 内容。
2. **判定结果**:
   - **完全符合**: 直接跳转至 [阶段 5: 审核通过]。
   - **部分符合**: 进入 [阶段 3: 建议与反馈]。

### 阶段 3：建议与反馈 (交互式)
1. **输出建议**: 列出具体的修改建议，按优先级排序（P0 阻塞项, P1 优化项）。
2. **询问用户**: 使用 `clarify` 工具询问：“您是否接受上述建议？”
   - 选项: `接受 (AI 补充)` / `拒绝 (我自行修改)` / `其他`
3. **分支处理**:
   - **拒绝**: 提示用户：“好的，请自行修改文档后告知我，我将重新审核。” -> **循环回 [阶段 1]**。
   - **接受**: 进入 [阶段 4: 内容补充]。

### 阶段 4：内容补充与确认
1. **生成补充内容**: 根据建议，生成修改后的 PRD 片段或完整更新版。
   - 🔴 **标记规范**：对于**新增或修改**的内容，必须使用 `🔴` 前缀或 `🔴 **...**` 格式进行高亮，以便用户快速识别变更点。
   - 展示给用户进行确认。
2. **询问确认**: 使用 `clarify` 工具询问：“补充/修改后的内容是否符合您的实际需求？”
   - 选项: `符合` / `需要微调`
3. **分支处理**:
   - **符合**: 跳转至 [阶段 5: 审核通过]。
   - **需要微调**: 进入 [阶段 4.1: 对话微调模式]。

#### 阶段 4.1：对话微调模式
- 提示用户：“请告诉我具体需要调整的地方。”
- 根据用户输入，实时更新内容并再次展示。
- 重复使用 `clarify` 询问确认，直到用户选择 `符合`。

### 阶段 5：审核通过
1. **输出结论**: 明确告知用户“审核通过”。
2. **交付物**: 
   - 如果进行了修改，提供最终版的 Markdown 文件路径或内容。
   - 简要总结本次审核的修改点。
3. **结束流程**。

## 注意事项
- **文档版本控制**: 所有修改建议基于本地副本，不直接覆盖飞书原文档，除非用户明确要求。
- **上下文保持**: 在微调循环中，始终维护最新的 PRD 状态。
- **标准占位**: 当前审核标准为通用模板，后续可通过 `references/` 注入垂直领域标准。
