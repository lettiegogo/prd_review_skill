# 产品文档审核工作流 (PRD Review Workflow)

本 Skill 定义了审核产品需求文档 (PRD) 的交互式工作流。核心目标是确保 PRD 符合标准，并通过对话循环完善文档。

## 触发条件
当用户提供飞书文档链接或 PRD 文件，并要求进行"审核"、"评审"或"检查"时触发。

## 工作流步骤

### 阶段 1：读取与预处理
1. **获取文档**: 🔴 **仅支持飞书文档链接**
   - 如果用户提供飞书链接：使用 `feishu-doc-export` skill 下载文档。
     - 🔴 **关键步骤**：飞书原始导出内容包含大量 XML-like 标签（如 `<lark-table>`, `<image token="...">`），**不可读**。必须使用 `feishu-doc-export` 下的 `scripts/clean_feishu_md.py` 脚本或等效的正则逻辑将其清洗为标准 Markdown，再进行后续处理。
   - 🔴 **检查阅读权限**：如果无法读取文档内容（权限不足），立即告知用户："我当前没有该 PRD 文档的阅读权限，无法获取文档内容。请您为我开通阅读权限后，我将重新执行审核流程。" 然后停止流程，等待用户确认。
   - 🔴 **不支持本地文档**：如果用户提供本地文件，请告知用户："本审核工作流仅支持飞书在线文档。请提供飞书文档链接，我将直接在线文档中完成审核与补充。" 然后停止流程，等待用户提供飞书链接。
2. **识别需求类型与加载标准**:
   - 分析 PRD 内容，判断需求类型：🔴 全新 (New) / 🟡 迭代 (Iteration) / 🟢 优化 (Optimization)。
   - 读取 `references/ops_web_standard.md` 并加载对应类型的审核章节：
     - **全新需求**: 关注权限、状态机、日志 (高严格度)。
     - **迭代需求**: 关注兼容性、旧数据、在途工单 (中严格度)。
     - **功能优化**: 关注变更范围、性能、回滚 (低严格度)。

### 阶段 2：初次审核与合规检查
1. **执行审核**: 对照标准检查 PRD 内容。
2. **判定结果**:
   - **完全符合**: 直接跳转至 [阶段 5: 审核通过]。
   - **部分符合**: 进入 [阶段 3: 建议与反馈]。

### 阶段 3：建议与反馈 (交互式)
1. **输出建议**: 列出具体的修改建议，按优先级排序（P0 阻塞项，P1 优化项）。
   - 🔴 **输出格式规范**：每条建议必须严格遵循以下视觉模板：
     - **标题**: `[编号]. [简短概括问题]`
     - **定位文档**: `- 📍 定位文档：「[标题序号] [标题名称]」`（例如：`「2.3 权限管理」`、`「4.1.2 状态流转」`）
     - **修改/补充内容**:
       - **修改场景**:
         - `- 原文：[引用原文内容]`
         - `- 🔴 修改后：[修改后的完整内容]`
       - **补充场景**:
         - `- 🔴 需补充：[补充内容描述]`
2. **询问用户**: 使用 `clarify` 工具询问："您是否接受上述建议？"
   - 选项: `接受 (AI 补充)` / `拒绝 (我自行修改)` / `其他`
3. **分支处理**:
   - **拒绝**: 提示用户："好的，请自行修改文档后告知我，我将重新审核。" -> **循环回 [阶段 1]**。
   - **接受**: 进入 [阶段 4: 内容补充]。

### 阶段 4：内容补充与确认
1. **生成补充内容**: 根据建议，生成修改后的 PRD 片段或完整更新版。
   - 🔴 **输出格式规范**：每条内容必须严格遵循以下模板：
     **[编号]. [问题标题]**
     - 问题描述：[简要说明修改/补充的原因]
     - 文档定位：[在「[标题序号] [标题名称]」章节...]（例如：`在「2.3 权限管理」章节...`）
     - 修改内容：[将原文修改为...]（如有）
     - 补充内容：[直接展示具体的补充文案]（如有）
   - 🔴 **注意**：修改内容和补充内容不一定同时存在，需根据问题类型灵活展示。
   - 展示给用户进行确认。
2. **询问确认**: 使用 `clarify` 工具询问："补充/修改后的内容是否符合您的实际需求？"
   - 选项: `符合` / `需要微调`
3. **分支处理**:
   - **符合**: 跳转至 [阶段 5: 审核通过]。
   - **需要微调**: 进入 [阶段 4.1: 对话微调模式]。

#### 阶段 4.1：对话微调模式
- 提示用户："请告诉我具体需要调整的地方。"
- 根据用户输入，实时更新内容并再次展示。
- 重复使用 `clarify` 询问确认，直到用户选择 `符合`。

### 阶段 5：审核通过
1. **输出结论**: 明确告知用户"审核通过"。
2. **询问回写**: 🔴 询问用户："是否将审核补充内容直接写入飞书原文档？"
   - 选项: `是 (写入原文档)` / `否 (仅展示结果)`
3. **分支处理**:
   - **否**: 在对话中展示最终审核结果和修改点，结束流程。
   - **是**: 进入 [阶段 5.1: 回写飞书文档]。

#### 阶段 5.1：回写飞书文档
1. **首次写入**：使用 `lark-cli docs +update --mode append` 将审核结果写入飞书原文档。
   - 🔴 **格式注意**：避免使用 Markdown 引用块 `>` 包裹补充内容，飞书解析引用块时容易吞掉嵌套列表和换行。改用 **加粗** + 普通列表格式。
   ```bash
   cat << 'MARKDOWN_EOF' | lark-cli docs +update --as user --doc "<doc_token>" --mode append --markdown -
   ## AI 审核补充内容
   ...（审核补充内容，使用加粗+列表格式，避免引用块）...
   MARKDOWN_EOF
   ```
2. 🔴 **写入后自查**：首次写入成功后，**必须立即重新获取文档内容**（`lark-cli docs +fetch`），验证刚写入的审核补充内容是否完整渲染。重点检查：
   - 嵌套列表是否完整显示
   - 引用块/代码块等格式是否被正确解析
   - 是否有内容被截断或丢失
3. **自查结果处理**：
   - **自查通过**（内容完整）：通知用户"审核补充内容已成功写入飞书文档，请查看：[文档链接]"，结束流程。
   - **自查发现问题**（内容有缺失/格式异常）：告知用户检测到写入不完整，将自动重试。先使用 `--mode delete_range --selection-by-title "## AI 审核补充内容"` 删除已写入的内容，再调整格式后重新写入，然后再次自查。
4. **用户反馈修复**：如果自查通过但用户反馈内容不完整（如第四条缺失），🔴 **必须先删除原有内容再重新写入**，避免重复追加：
   ```bash
   # 第一步：删除已有的审核补充内容
   lark-cli docs +update --as user --doc "<doc_token>" --mode delete_range --selection-by-title "## AI 审核补充内容"
   # 第二步：调整格式后重新完整写入
   cat << 'MARKDOWN_EOF' | lark-cli docs +update --as user --doc "<doc_token>" --mode append --markdown -
   ## AI 审核补充内容
   ...（完整内容）...
   MARKDOWN_EOF
   ```
5. 🔴 **内容一致性要求**：写入飞书文档的内容必须与对话中展示给用户确认的内容在结构和信息上**完全一致**，包括格式和排版。对话中使用的 `🔴` 前缀仅用于对话高亮，**写入飞书文档时不需要保留 `🔴` 前缀**，保持文档内容整洁即可。
6. **分支处理**：
   - **如果写入成功且自查通过**: 简要总结本次审核的修改点，结束流程。
   - **如果写入失败（权限不足）**：🔴 立即告知用户："我当前没有该 PRD 文档的编辑权限，无法将内容写入原文档。请您为我开通编辑权限后，我将重新执行写入。" 然后停止流程，等待用户确认。
   - 🔴 **禁止行为**：❌ 不要写入本地文档作为替代。❌ 不要跳过权限检查继续执行。❌ 不要在已有审核内容的情况下直接 append（会导致重复），必须先删除再写入。

#### 飞书写入格式注意事项
- 🔴 **禁止在写入飞书文档的内容中使用 `>` 引用块格式**：飞书的 Markdown 解析器会吞掉引用块内的内容，导致补充内容丢失。务必使用 **加粗 + 列表** 格式替代，例如用 `**人群构建异常处理：**` + `- 构建失败时...` 替代 `> **人群构建异常处理：**\n> - 构建失败时...`。
- 写入前检查：如果补充内容中包含引用块，必须在写入时转换为普通列表/加粗格式。

## 注意事项
- 🔴 **仅支持飞书文档**：不支持本地文件审核，必须提供飞书在线文档链接。
- 🔴 **禁止写入本地文档**：所有修改必须直接写入飞书原文档（需用户确认回写且具备编辑权限），如果权限不足，必须要求用户开通权限，不得以本地文档作为替代方案。
- **上下文保持**: 在微调循环中，始终维护最新的 PRD 状态。
- **标准占位**: 当前审核标准为通用模板，后续可通过 `references/` 注入垂直领域标准。
